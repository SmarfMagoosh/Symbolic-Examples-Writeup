\section{Introduction} \label{sec:introduction}
\textit{Property-based testing}
is an approach to testing programs.
In this approach,
a programmer identifies
\textit{properties}, or specifications,
of functions or groups of functions,
and those properties are then tested on
a large number of randomly selected values.
Property-based testing was first popularized in
the functional programming ecosystem,
with the Haskell library QuickCheck~\cite{claessen2000quickcheck}
being a particularly influential work.
However, there are now property-based testing libraries
for many other languages, both imperative and functional,
such Python~\cite{maciver2019hypothesis},
Erlang~\cite{arts2008testing},
and
Rust~\cite{derbenwick2025property}.

Property based tests
often ensure that two expressions satisfy some kind of relation
(often, equality)
when applied to similar inputs.
For example,
consider the \hskin{zip} function defined in
Figure~\ref{fig:zip-code}.
This function ``zips'' two lists together
by putting each corresponding pair of values into a tuple.
If one list is longer than the other, the longer list is cut off.
To apply property-based testing,
one would aim to find (expected to be true)
properties of \hskin{zip},
and then test those properties.
For example, one can say that for all \hskin{xs} and \hskin{ys},
we have that \hskin{length (zip xs ys) <= length xs}
and \hskin{length (zip xs ys) <= length ys}.
Perhaps more interestingly, one can also write something like
\hskin{map fst (zip xs ys) == take (length ys) xs},
which (roughly) says that zipping \hskin{xs} and \hskin{ys}
preserves the values and orders of the first \hskin{length ys} elements in \hskin{xs}.

Property-based testing is incredibly useful for testing,
as evidenced by its proliferation to a wide array of language.
However,
both writing and understanding properties can also be challenging,
as it requires determining how various functions in a codebase interrelate.
In this paper,
we propose a specialized DSL \ourDSL
which allows writing property-based unit tests, focused on a single function
(mainly, with the exception of some necessary use of list conversion functions.)
Once a programmer understands the language,
the DSL allows writing tests for particular functions independently of
other program code.
This also has implications for other programmers, who might be trying to read the code later:
tests written in \ourDSL can be viewed as a sort of testable specification,
and therefore aid in \textit{understanding} (wholly) unfamilar code.

\mypara{Example}
Figure~\ref{fig:zip-test}
shows a test of \hskin{zip} written in \ourDSL.
The DSL allows writing \textit{symbolic expressions}
to specify expected inputs and outputs of functions.
As \hskin{zip} takes two inputs,
we apply a function \hskin{test2}
to \hskin{zip}.
We also pass two symbolic expressions (line~\ref{lne:zip-input})
specifying \hskin{zip}'s input.
The first of these arguments
uses a regular expression-like Kleene closure \hskin|x*{n}| to build a list.
However, unlike in a traditional regular expression,
the Kleene closure is annotated with a label \hskin|{n}| denoting that the pattern must be matched \hskin{n} times.
This gives us a list of length \hskin{n}.
On each unrolling, a \textit{new} value for \hskin{x} is chosen,
hence, we essentially end up with an arbitrary list.
The second argument is specified as \hskin|y*{n}|,
giving us a different list with randomly generated elements,
also of length \hskin{n}.

We then specify the output,
\hskin|(<(x, y)>)*{n}|.
This expression says we get a list of tuples
of length \hskin{n},
where the first element of the tuple (\hskin{x}) is
from the first input list
 and the second element of the tuple (\hskin{y}) is from the second input list.
The unrollings of the stars are synchronized,
such that we know that the $i^{th}$ value of \hskin{x} (resp. \hskin{y}) in the input list
must be equal to the $i^{th}$ value of \hskin{x} (resp. \hskin{y}) in the output.
We thus have a precise specification for \hskin{zip},
in the case where the input lists are the same length.

\mypara{Contributions}
In short, our contributions are as follow:
\begin{itemize}
\item A novel design for a property-based testing library,
in which an eDSL is used to specify inputs and outputs
as \textit{symbolic expressions}.
We define formal rules for both generating values from,
and checking conformance of outputs to,
expressions in the eDSL.

\item Discussion of implementation techniques
for the eDSL in a statically typed language (in particular, we consider Haskell.)
Due to the need to ensure that all generated expression typecheck,
this is non-trivial, requiring either a somewhat complex encoding or unsafe coercing of values (we describe the former.)

\item An implemention of the eDSL, called \ourDSL,
and a set of case studies,
in which we apply property-based tests
written with \ourDSL to a variety of data structure libraries.
\end{itemize}
