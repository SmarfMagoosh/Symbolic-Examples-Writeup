\section{Introduction} \label{sec:introduction}
\textit{Property-based testing}
is an approach to testing programs.
In this approach,
a programmer identifies
\textit{properties}, or specifications,
of functions or groups of functions,
and those properties are then tested on
a large number of randomly selected values.
Property-based testing was first popularized in
the functional programming ecosystem,
with the Haskell library QuickCheck~\cite{claessen2000quickcheck}
being a particular influential work.
However, there are now property-based testing libraries
for many other languages, both imperative and functional,
including libraries for Python~\cite{maciver2019hypothesis},

Property based tests
typically ensure that two expressions satisfy some kind of relation
(often, equality)
when applied to similar inputs.
For example,
consider the \hskin{zip} function defined in
Figure~\ref{fig:zip-code}.
This function ``zips'' two lists together
by putting each corresponding pair of values into a tuple.
If one list is longer than the other, the longer list is cut off.
To apply property-based testing,
one would aim to find (expected to be true)
properties of \hskin{zip},
and then test those properties.
For example, one can say that forall \hskin{xs} and \hskin{ys},
we have that \hskin{length (zip xs ys) <= length xs}
and \hskin{length (zip xs ys) <= length ys}.
Perhaps more interestingly, one can also write something like
\hskin{map fst (zip xs ys) == take (length ys) xs},
which (roughly) says that zipping \hskin{xs} and \hskin{ys}
preserves the values and orders of the first \hskin{length ys} elements in \hskin{xs}.

Property-based testing is incredibly useful for testing,
as evidenced by its proliferation to a wide array of language.
However,
both writing and understanding properties can also be challenging,
as it requires determining how various functions in a codebase interrelate.
In this paper,
we propose a specialized DSL \ourDSL
which allows writing property-based unit tests, focused on a single function.
Once a programmer understands the language,
the DSL thus allows writing tests for particular functions independently of
other program code.
This also has implications for other programmers, who might be trying to read the code later:
tests written in \ourDSL can be viewed as a sort of testable specification,
and therefore aid in \textit{understanding} (wholly) unfamilar code.
