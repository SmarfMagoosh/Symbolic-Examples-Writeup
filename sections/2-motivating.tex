\section{Motivating Example} \label{sec:motivating-example}

\begin{figure}
\begin{smallhskcode}
zipTest = test2 "zip" (zip :: [Int]->[Int]->[(Int, Int)])
        [ex| x*{n}(.* |{b} []) |] (* \label{lne:comp-zip-input1} *)
        [ex| y*{n}([] |{b} .*) |] (* \label{lne:comp-zip-input2} *)
    .== [ex| <(x, y)>*{n} |]                     (* \label{lne:comp-zip-out} *)
\end{smallhskcode}
\caption{A complete \hskin{zip} test in \ourDSL}
\label{fig:zip-complete-test}
\end{figure}

In the introduction,
we showed a test for \hskin{zip}
that generated pairs of list arguments,
where both lists were the same length.
This test is partial--
it tests \hskin{zip} on a subset of its domains
(namely, pairs of lists that are the same length.)
Figure~\ref{fig:zip-complete-test}
shows a complete test for \hskin{zip}
that accounts for the entire domain.
The symbolic expression for the first list,
shown on line~\ref{lne:comp-zip-input1},
uses \hskin|x*{n}| to generate $n$ inputs,
as before, but then follows that with
a union between an arbitrary suffix and an empty list,
\hskin+(.* |{b} [])+.
The second list, on line~\ref{lne:comp-zip-input2},
is also augmented to allow a suffix,
\hskin+([] |{b} .*)+.
Notice that, just as the Kleene closures have a label
\hskin{n} that links them together,
the unions also have a label \hskin{b}.
This forces both expressions to match the same side.
Thus, one of the lists will be of length \hskin{n},
and the other may be arbitrarily longer.
The output symbolic expression is unmodified from
the previous \hskin{zip} test,
reflecting the fact that the shorter input has length \hskin{n}
also means that the output list will have length \hskin{n}.

We now consider how these symbolic expressions
are used to test \hskin{zip}.
\ourDSL performs tests in 2 phases; the generation phase and the validation phase.
During the generation phase, it uses the expressions for the arguments to generate inputs for the function.
During the validation phase, it checks if the expression for the output matches the value returned by running the function on the generated inputs.

\mypara{Generation Phase} 
Firstly, \ourDSL generates a value from the first symbolic expression \hskin+[ex| x*{n}(.* |{b} []) |]+.
When it sees \hskin+x*{n}+,
the value for \hskin{n} is not specified,
thus it generates a random value for \hskin{n}.
We then must repeat the pattern \hskin{x} a total of \hskin{n} times.
On each iteration, a new 
random value for \hskin{x} is generated.
These random values are then concatenated into a list.
When each value of \hskin{x} is generated, \ourDSL remembers
the value it picked for \hskin{x} for each value of \hskin{n}.

% and all values in a list. 
For example, suppose \ourDSL randomly chooses $n=2$.
It will then generate two random values for \hskin{x}:
for instance, $x=1$ and then $x=5$. We will then store an environment
(which also contains the value for \hskin{n} from before):
\begin{align*}
\{ x &\rightarrow (n = 0 \rightarrow 1; n = 1 \rightarrow 5), \\
   n &\rightarrow 2 \}
\end{align*}
\ourDSL has now used \hskin|x*{n}| to generate the value
\hskin{[1, 5]}.
\ourDSL now moves on to the next part of the symbolic expression,
\hskin+(.* |{b} [])+.
% so that \code{x*\{n\}} generates the value \code{[1, 5]}.
To choose which branch of the union to follow,
\ourDSL will then generate a random value for \hskin{b}.
Based on the parity,
we will then follow either the left branch (for even values)
or the right branch (for odd values.)
Suppose $b = 1$ is randomly chosen.
This value for \hskin{b} will be stored in the environment,
and will lead to picking the right hand side of the union,
giving us an empty list.
This is then concatenated to our list from the first Kleene closure,
to give us a final input of \hskin{[1, 5]}.

% Then it will generate a value for \code{[] |\{b\} .*} and concatenate the generated value to the last result.
% For a labelled union, it will generate a value and, based on the parity, generate a value from the left or right hand side.
% Suppose we get $b=0$, it would then generate the left hand side which just gives us the empty list.
% Now it concatenates to get the final result of \code{[1, 5] \concat [] = [1, 5]}.

After generating the value for the first expression, it repeats the process for the second expression; \hskin|y*{n}([] |{b} .*)|.
Since \hskin{n} and \hskin{b} were already assigned the value in the previous expression, those values are reused in this expression. 
Two values for \hskin{y} are chosen as
a list is generated by \hskin|y*{n}|,
for instance, $y = 4$, followed by $y = 6$.
At this point, we will add these entries to the environment:
\begin{align*}
\{ x &\rightarrow (n = 0 \rightarrow 1; n = 1 \rightarrow 5), y \rightarrow (n = 0 \rightarrow 4; n = 1 \rightarrow 6), \\
  n &\rightarrow 2, b \rightarrow 1 \}
\end{align*}
\ourDSL will then go on to generating values
from the second half of the symbolic expression
\hskin+([] |{b} .*)+.
Because \hskin{b} has been assigned the value \hskin{1},
the right hand side will be evaluated,
allowing generation of an arbitrary list,
$[8, 3]$ for example.
% So all it has to do is generate a value for $y$ 2 times; $y_1=4$, $y_2=6$. 
% For the second half, we generate value from \code{.*} since that's on the left hand side of the union.
% The expression \code{.*} just generates a random value a random amount of times as a list, \code{[8, 3]} for instance.
This gives us the final result of \hskin{[4, 6] ++ [8, 3] = [4, 6, 8, 3]}
The final step of the generation phase is to apply the function
being tested
to the generated arguments, 
\hskin{zip [1, 5] [4, 6, 8, 3] = [(1, 4), (5, 6)]}
This result gets passed to the validator for phase 2.

\mypara{Validation Phase}
Now that \ourDSL has computed an output from random inputs, 
it must verify that it is possible for the output symbolic expression to generate the computed value passed from the generator.
Any values assigned to labels
(which we also refer to as \textit{variables})
from the inputs get carried over to the output.
We start with $n=2$ from the previous expressions, 
so \ourDSL must match the internal expression, \code{<(x, y)>} twice to validate the output.
The angle brackets indicate that the value is a data constructor applied to some arguments,
in this case, the tuple constructor applied to \hskin{x} and \hskin{y}.
According to our environment, when $n=0$ we have $x=1$ and $y=4$ 
so $(x, y)=(1,4)$.
Since the value from the expression is equal to the first element of our computed list, \ourDSL concludes that there is a match.
\ourDSL consumes the matched portion of the input and moves on to the next iteration of the Kleene closure where $n=1$.
We now have, $x=5$ and $y=6$ so $(x, y)=(5, 6)$ which is the second element of our computed list.
As this is another match, we consume this value and continue.
Now that we've exhausted the symbolic expression since $n=2$ and \ourDSL has
matched twice, we check to see if the computed output has been fully consumed
as well. Since it has, we say the test was successful.
