\section{Related Work}

\mypara{Property-Based Testing}
In this work, we consider a new approach to property-based testing
based on writing function specifications in a \textit{symbolic expressions} eDSL.
We explore this direction in the functional programming world,
where in particular QuickCheck~\cite{claessen2000quickcheck},
alongside other approaches such as SmallCheck~\cite{runciman2008smallcheck}
and Target~\cite{seidel2015type},
has led to property-based testing becoming a popular paradigm.

QuickCheck~\cite{claessen2000quickcheck}, a Haskell property-based testing library, is often credited as popularizing
property-based testing and inspiring
property-based testing libraries~\cite{maciver2019hypothesis,benac2014jsongen,denes2014quickchick,seidel2015type}.
However,
the earliest work on property-based testing~\cite{fink1994property,fink1997property}
was actually on C code in the Unix community,
where property based testing was used to check security properties and search for exploits.
It introduced a specialized language, TASPEC, for writing testing oracles and specifications,
with an eye specifically toward security.

Despite this, the statement that QuickCheck~\cite{claessen2000quickcheck},
was highly influential on much, if not all,
of the work that follows is quite fair.
QuickCheck allows writing specifications over functions using mostly ordinary Haskell code,
using typeclass instances to specify random input generation.
SmallCheck~\cite{runciman2008smallcheck} and Target~\cite{seidel2015type}
are alternative Haskell property-based testing library.
SmallCheck closely resembles QuickCheck,
except that
functions are tested on inputs of increasing size.
Target tests \textit{refinement types}
i.e. regular types annotated with decidable predicates.
This refinement types can also be verified with the LiquidHaskell~\cite{vazou2014refinement} verifier;
a major motivation of Target is to act as a complementary tool to LiquidHaskell.

Adaptions of the core ideas of QuickCheck
exist for a wide variety of other languages, both functional and imperative;
as a (much abbreviated) list,
we note QuickCheck for Isabelle~\cite{bulwahn2012new},
QuickCh\textit{i}ck for Rocq~\cite{denes2014quickchick},
QuickCheck for Rust~\cite{derbenwick2025property},
Hypothesis for Python~\cite{maciver2019hypothesis},
PyModel for Python~\cite{jacky2011pymodel},
Quviq QuickCheck for Erlang~\cite{arts2008testing},
and 
Jsongen for json web services~\cite{benac2014jsongen}.

\mypara{Symbolic Expressions}
Symbolic expressions can be viewed as mini-programs
that generate (related) function inputs and outputs.
Alternatively, they can be seens as abstractions or generalizations
of the inputs themselves.
This second view,
% of the expressions as abstractions,
suggests a relation to the manipulation of symbolic program values performed
by program analysis tools.
The most direct link to our work
in this domain is synthesis with abstract examples~\cite{drachsler2017synthesis}.
The ``abstract examples'' in question
represent
strings using a regular expression like language,
extended with labeled kleene stars
and variables.
Unlike \ourDSL, these examples are used purely as an \textit{output mechanism}--
the synthesizer asks the user if certain abstract example input/output pairs
correspond to the function being synthesized, and the user answers yes or no.
The abstract example language in~\cite{drachsler2017synthesis}
is less sophisticated than ours;
kleene stars are only allowed to repeat a single character,
variables correspond to a single character,
and there are no further features
(i.e. no labeled unions or nested lists.)
Kleene algebra with tests (KAT)~\cite{kozen1997kleene}
is a generalization of regular expressions
which has been applied to network program verification~\cite{anderson2014netkat}.
The generalization is rather different than ours;
KAT lacks any equivalent of labels, for example.

Other uses of symbolic representations are further removed,
but still worth considering.
In the world of functional programming,
symbolic execution tools such as
Rosette~\cite{torlak2014lightweight}, G2~\cite{hallahan2019lazy}, SmtEn~\cite{uhler2014smten}, and CutEr~\cite{giantsios2015concolic}
exploit such reasoning to support finding program bugs and (in some cases) verify programs.
These tools general reserve symbolic representations for \textit{internal} use,
communicating to the user by generating corresponding concrete values.
