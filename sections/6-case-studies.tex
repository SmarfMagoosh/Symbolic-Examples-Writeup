\section{Case Studies}

In this section, we discuss some case studies in which we applied
\ourDSL to a variety of data structures.
To do this we collected real-world code
from multiple sources:
we used 25 functions from \hskin{Data.List}
(part of GHC's standard library)
 14 functions from \hskin{Data.Map} (part of the widely used \texttt{containers} package)
and 3 from a \hskin{Union-Find} data structure
implemented as part of an academic Haskell project~\cite{hallahan2019lazy}
following the basic design of~\cite{conchon2007persistent}.

\mypara{List}
We wrote 25 \ourDSL tests for \texttt{Data.List},
each involving a single function from the library.
The \hskin{zip} test, from the motivating example (Section~\ref{sec:motivating-example}),
is one example of a test for this library.
A second example, for \hskin{++} (append),
is shown in Figure~\ref{fig:list-append-ex}.
% Here, \hskin{test2} acts as a higher-order function, taking four arguments: 
% (1) the function to test, (2) a symbolic expression for its first argument, 
% (3) a symbolic expression for its second argument, 
% and (4) a symbolic expression for the expected output.  
In the \hskin{++} example, the first symbolic example, \hskin;[ex| b*{a} |];,
is the core expression for the 
first input to \hskin{(++)}, generating a random list of integers of length a. 
Similarly, the third argument \hskin;[ex| d*{c} |]; generates a random list 
of integers of length c. Lastly, the symbolic expression \hskin;[ex| b*{a} d*{c} |]; 
stands for the concatenated output of these two lists. 
\ourDSL generates random inputs for input symbolic expressions and 
runs the \hskin{(++)} with those inputs. 
It then compares the output against the output symbolic expression. 

\mypara{Higher-order functions}
\ourDSL currently does not possess a way to generate arbitrary functions, so testing a high order function on arbitrary inputs is impossible. 
However, it is possible to test high order functions through partial application. 
For example, consider the test in Figure~\ref{fig:list-filter-ex}.
The filter function accepts a predicate as its first argument and a list as its second, and returns a list containing only the elements where the predicate is true.
To test this with
\ourDSL,
we apply \hskin{filter}
in a lambda \hskin{\x -> filter (/= x)}.
This fixes filter to removing some specific element,
but allows the symbolic test to vary that element,
and vary the actual list that is passed in.
We can then specify precisely what the output
should be relatively to the input:
we use a labeled union to
select including either an \hskin{x}
or a not-equal-to-\hskin{x} element in the list,
\hskin+(x |{b} (y /= x))*{n}+.
We then can specify
that in the output list, all the \hskin{x}s
should be missing, and all other elements should be present,
\hskin+([] |{b} y)*{n}+.
% This is obviously not a proper high order function since none of the arguments are functions themselves,
% however we can still test the implementation of a high order function, filter in this case, by predetermining what the function arguments should be
% and accepting extra arguments that we can randomly generate to construct them.

\mypara{Map} \label{subsec:case-studies-map}
To test the effectiveness of \ourDSL with other data structures, 
we also evaluated it on functions from Haskell's \hskin{Data.Map} 
library. We selected Map because symbolic expressions in our 
approach can generate only lists, and \hskin{Map} provides functionality 
to convert between maps and lists, making it suitable for our tests.

Figure~\ref{fig:lookup-test} shows an example test for a function 
\hskin{lookup} of \hskin{Map} data structure. Here, the function 
that we are passing to \hskin{test2} is \hskin{(\x -> lookup x . fromList)} 
of type \hskin{Int -> [(Int, Int)] -> Maybe Int}. The original 
type definition of \hskin{lookup} is \hskin{k -> Map k a -> Maybe a} 
where it takes a key and a map, looks up for the key of some type 
\hskin{k} and returns the value of type \hskin{a} wrapped in a 
\hskin{Maybe}. It returns \hskin{Nothing} if the key does not exist 
in the Map or returns the value by wrapping it in a \hskin{Just} constructor.

Since we only generate lists from symbolic expressions, during function 
application, we convert the \hskin{Map} to \hskin{List} using the 
\hskin{fromList} function. The expression \hskin;[ex| k |]; generates a random key. 
In expression \hskin;[ex| <((. /= k), .)>* <(k,v)> <((. /= k), .)>* |]; 
one might notice the angled brackets(<>), which denote data 
constructor application. Since a map can be represented as a list of tuples,
where \hskin{(,)} is a constructor that carries two values, 
we need a data constructor application. So, in the subexpression 
\hskin{<((. /= k), .)>* } angled brackets perform \hskin{(,)}
constructor application. Within the same sub-expression, \hskin{(. /= k)}
is actually a predicate that generates a key which is not equal to the one 
we are looking for. Together \hskin{<((. /= k), .)>* }
generates the list of key-value pair 
that does not contain key \hskin{k}. Sub-expression \hskin{<(k, v)>} 
generate a tuple \hskin{(k,v)}. Later, the combined result of all the 
sub-expressions forms the input list. Forming the symbolic 
expression in the mentioned way, make sure the input 
list contains only one such key-value pair.

\mypara{Union-Find}
We further tested \ourDSL on a union-find data structure,
extracted from an existing academic project~\cite{hallahan2019lazy},
and (based on a comment in the code) based on the design from~\cite{conchon2007persistent}.
Union-Find is a specialized data structure for representing a collection of disjoint sets, 
with each set being assigned one element to represent it.
The Union-Find data structure has two primary operations that it can perform, union and find.
The union operation takes 2 elements of the structure as parameters, finds the two sets that they are in, and unions them together.
The find operation takes 1 element as a parameter, and returns the representative from the set that contains it.
For example, you can determine if two elements are in the same set by comparing the corresponding representatives returned by the find operation.

\par In Figure~\ref{fig:uf-union-test} we see a symbolic test for the union operation. Similarly to with Maps, we must
use \hskin{toList} and \hskin{fromList} to convert the data structure we
are actually testing to lists that can be generated by \ourDSL.
As lists, union-finds are two dimensional. Each element is one of its
non-singleton disjoint sets that it maintains. 
For example, if after initializing a union-find the operations \hskint{union 1 100} and \hskin{union 5 7} were performed, the lists representation would be \hskin{[[1, 100], [5, 7]]}.
The test above shows a function taking two integer inputs that are not equal. 
The third parameter takes a two dimensional list representing a union find. It is an arbitrary set containing \hskin{x} and an arbitrary set containing y, with any number of arbitrary sets between and around them. The output shows \hskin{x} and \hskin{y} in the same set, having been unioned together.

\par Since the output contains many dots, we call this a partially specific test, in that while certain properties of the output are known, namely \hskin{x} and \hskin{y} being in the same set, the exact value of the output is not entirely clear.

\subsection{Threats to Validity}

\mypara{Internal Validity}
The primary limitation of \ourDSL is its inability to reference label's outisde of its current evaluation context.
In other words, if it is working with the expression \hskin|x*{y}|, 
then it only ever has access to one value for $x$ which is determined by the context of $y$. 
This makes it impossible to consider permutations of \hskin|x*{y}|, find the minimum $x$ generated, or determine if there are duplicate $x$ values.
These issues caused significant problems when writing tests for \hskin{Data.Map} since we couldn't ensure uniqueness in the key sets or account for
the fact that converting the map to the list might change the order of the tuples around.

\mypara{External Validity}
The reason \ourDSL works so well in Haskell is because Haskell has several features;
GADTs, type level functions, and QuasiQuotations; that allow us to
to a lot of preprocessing during compilation to help catch errors faster.
While these features aren't very common to have in other programming languages,
these other languages will usually have a way to get around that. For example in a 
dynamically-typed language like Python, it would be impossible to tell if a symbolic expression is well formed before runtime.
At the same time however, its unnecessary to check anything before runtime since Python's philosophy is to keep running until
something forces it to crash. This means that \ourDSL could be implemented in a language like Python its just that most of the
compile time errors we get in Haskell would be runtime errors instead. We could achieve similar results in statically typed languages
such as Java via type casting and proper error handling with try-catch blocks.
