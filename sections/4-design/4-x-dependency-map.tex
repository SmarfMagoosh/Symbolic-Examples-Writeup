\subsection{\bill{Mappings}}
\label{sec:mappings}
Before getting to the actual rules for generation and validation,
we must introduce three key constructs used in these rules.
These are the \textit{dependency map},
the \textit{path},
and the \textit{environment.}
These three constructs work together
to allow us to track values of labels in symbolic expressions.
Note that,
in a symbolic expression,
the value of a variable may depend on the value of surrounding variables.
In particular, this happens when a variable is assigned inside a Kleene closure.
For example the expression $a \asgnarrow (b \asgnarrow b)*_{a}$ will match any list of elements. 
The elements won't necessarily all be equal even though they were all matched against $b$,
because $b$ is bound within the Kleene closure,
and is allowed to take on different values depending on which iteration $a$ we are on.
Thus the value of $b$ \textit{depends on} $a$.
% This is why we use the context so that $b$ can take on a different value for each iteration of $a$.

\mypara{Dependency Map} 
% In a symbolic expression we sometimes see variables defined in the context of other variables.
% This primarily happens when a variable is declared inside a Kleene closure. 
% However, if $b$ is referenced without $a$ being in context, it becomes ambiguous which $b$ should be used.
To track dependencies between variables, we introduce the \textit{dependency map} $\deps$. 
The dependency map is a mapping that keeps track of which variables depend on which other variables. 
In the example above, $a$ doesn't depend on anything, and $b$ depends on $a$. 
This means that it is illegal to reference $b$ unless $a$ is in the current context--
such illegal accesses are rejected with an error.

\textit{Dependency Map Computation}
The dependency map is computed for a group of symbolic expressions that represent a function's
inputs and outputs \textit{once},
before we begin generating or checking values. 
The computation traverses the first expression keeping track of a \textit{context} \depscontext
of all Kleene star labels that it has passed through. 
When it comes across an assignment for some variable $\lab$, it adds an entry
pointing from the label to the current context $\deps[\lab \rightarrow \depscontext]$.
Once it finishes traversing the first tree, it has a dependency map for the first tree but not the whole function. So it
threads the result from the first traversal and repeats the process with the subsequent expressions.

\textit{Dependency Map Use}
The dependency map is used for environment lookups and updates. 
First, recall that every key in the environment is a context-label pair that gets mapped to a value.
So whenever a lookup or update occurs, the first thing that happens is we verify that all the label's
depencies are satisfied by the context that was passed in. Then we filter out all variables from the
context that are not in the label's dependencies. This gives us $\pathvar'$ which contains only the
relevant context for the label. This relevant context is what is actually used to lookup values and
insert pairs into the environment.

\mypara{Path}
% $\pathvar$ is the \textit{path} and $\env$ is the \textit{environment}; we elaborate on each of these in the following. 
% Understanding the path requires noting that, in generating lists (or, when we come to it later in Section~\bill{whatever}, validating lists) 
% we may repeatedly use an expression via a kleene closure, i.e. given $r*_\lab$,
% the expression $r$ may be repeated an arbitrary number of times. 
The path is a set of label-to-int mappings that tracks
the current values assigned by Kleene closures during either generator or validation.
On first entering a kleene closure $r*_\lab$,
$\lab \rightarrow 1$ is entered into the path.
On each successive $n^{th}$ iteration, this mapping is updated to $n$.

\mypara{Environment}
The environment tracks how generation is assigning labels to values via bindings,
$(lab \Longleftarrow p(args, \lab)) \mapsto e$
however note that the same label may be re-assigned different values in each iteration of a
(possibly nested) kleene closure,
thus, the environment is really a mapping of (label, path) pairs to values.
We write $\env^\deps_\pathvar(\lab)$
to denote looking up the label \lab in \env,
given some path and dependency map
(because \deps is always the same for any given symbolic expression,
we omit it in future sections to reduce syntactic clutter.)
The way this works is that
 vwe filter out all variables from the
in the path that are not in the \lab's dependencies.
This then gives us a $\pathvar'$ which contains only the
relevant context for the label,
and can be used for the lookup in the environment.

\begin{example}
\bill{Incorporate dependency map into example}
Consider the expression:

\[
(n \Longleftarrow \mathit{True}) \mapsto ((x \Longleftarrow \mathit{True}) \mapsto x)*_n
\]

One list that could be generated from this would be:

\[
[8, 4, 17]
\]

The corresponding environment would be:
\begin{align*}
    \{  &(n, \{\}) \rightarrow 3, \\
        &(x, \{ n \rightarrow 1 \}) \rightarrow 8, \\
        &(x, \{ n \rightarrow 2 \}) \rightarrow 4, \\
        &(x, \{ n \rightarrow 3 \}) \rightarrow 17 \\ 
    \}
\end{align*}

$n$ occurs outside of all kleene closures, and thus has an empty context.
$x$ is chosen to be $8$, $4$, and $17$, respectively,
on the $1^{st}$, $2^{nd}$ and $3^{rd}$ iterations of the unrolling of the kleene closure indexed by $n$,
and thus has a separate value for each of these three contexts.
\end{example}
