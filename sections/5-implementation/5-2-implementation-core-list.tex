Returning to the type of \hskin{test2},
we will focus on \hskin{Core n a} and \hskin{NestedList n a}.
A definition for the \hskin{NestedList} and a partial definition of \hskin{Core} can be seen in figure~\ref{fig:nestedlist}.
% In fact, \hskin{Core n a} is a GADT, and \hskin{NestedList n a}
\ourDSL uses these to create a correspondence between the types generated by core expressions
and the type of the actual function that will be tested.
Core expressions are GADTs of the form \hskin{Core (n :: Nat) a},
where $n$ is the \textit{depth} of the nested lists surrounding values of type $a$.
For example, the type \hskin{[[Int]]} would be represented by \hskin{Core 2 Int}.
\hskin{NestedList n a} is a type family
that computes the type that a \hskin{Core n a} would generate.
For example, \hskin{NestedList 2 Int} reduces to \hskin{[[Int]]}.
% A \code{Core n a} generates a \code{NestedList n a} where $n$ and $a$ have the same values as the expression that generated it.
The correspondence between these two allows generated values to be passed to functions so that the generator can compute the output of
    the testing function before handing it to the validator.
The generator function takes several arguments besides the core expression,
but a (simplified) view of its type would be:
\begin{smallhskcode}
generate :: Core n a -> NestedList n a
\end{smallhskcode}
We take a core expression that is supposed to produce an expression that is \hskin{n}-deep nested lists around \hskin{a}'s,
and the return type of \hskin{NestedList n a} says that this is indeed what is produced.
An (again, simplified) view of \hskin{validate} would be:
\begin{smallhskcode}
validate :: NestedList n a -> Core n a -> NestedList n a
\end{smallhskcode}
we take a \hskin{NestedList n a} and attempt to validate it against the \hskin{Core n a}--
then, we return any suffix of the list that is left over.

\par To see how this all fits together, consider the(trivial) test 
\begin{smallhskcode}
test2 ((:) :: Int -> [Int] -> [Int])
    [ex|x] [ex|y*{z}|] .== [ex|x y*{z}|]
\end{smallhskcode}
where \hskin{(:)} is Haskell's list cons constructor.
Specialized to lists of \hskin{Int}s, \hskin{:} has type \hskin{Int -> [Int] -> [Int]},
which can also be written using the \hskin{NestedList} type family as \hskin{NestedList 0 Int -> NestedList 1 Int -> NestedList 1 Int}.
\hskin;[ex|x]; is a lone variable,
as seen in the type of the \hskin{Core} constructor \hskin{Var}, this means it has type \hskin{Core 0 Int}.
\hskin;[ex|y*{z}|]; is a list, and has type \hskin{Core 1 Int}.
Thus, these align correctly with the expected argument type \hskin{NestedList 0 Int} and \hskin{NestedList 1 Int}.
Thus, calls to \hskin{generate} within \hskin{test} can turn the passed \hskin{Core} arguments
into \hskin{NestedLists}, which can then be passed to the higher-order function parameter to \hskin{test}.
The output is then guaranteed to be a \hskin{NestedList 1 Int} which can be compared to the expected output \hskin;[ex|y*{z}|];,
which has type \hskin{Core 1 Int}.

If the core expressions were not depth indexed (i.e if we had \hskin{Core a} instead of \hskin{Core n a}), 
then when the generator made a value
% \hskin{x} or \hskin{y*{z}},
the typechecker would have no way of knowing it can pass these value as the arguments of \hskin{(:)}
in a typesafe way (all arguments would simply have type \hskin{Core Int}).
Thus, the use of GADTs and type families is essential to get compile time error messages
from incorrect arguments to tests.
